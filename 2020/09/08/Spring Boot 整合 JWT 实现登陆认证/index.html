Spring Boot 整合 JWT 实现登陆认证

发表于 2020-09-08 | 分类于 Spring Boot

JWT 英文名是 Json Web Token ，是一种用于通信双方之间传递安全信息的简洁的、URL安全的表述性声明规范，经常用在跨域身份验证。

传统 Session 认证

Session

传统的 Java Web 项目中，我们通常通过 session 来处理登陆与认证逻辑。我们知道，HTTP 是一种无状态的协议，对于每一次请求，服务端无法知道该请求来自于哪一个客户端，由此便引入了“会话技术”（有 session 和 cookie）。会话中可以储存用户信息，一次会话即客户端与服务端之间进行多次请求（request）与响应（response）的过程，服务端从会话中读取用户信息，这样便可以知道每一次请求来自何处。

认证流程

基于会话技术，便有了如下通过 Session 来进行用户登陆与认证的流程：

  1. 客户端将用户输入的用户名与密码发至服务端；
  2. 服务端比对数据库，确认信息通过验证后，在当前 Session 中保存用户的相关信息，如用户名，用户权限等；
  3. 返回一个 session_id，并保存至客户端的 Cookie；
  4. 之后的每一次请求，客户端都会将 session_id 通过 Cookie 带往服务端；
  5. 服务端获取 Session ，从中取出先前保存过的用户信息，便可以得知该用户的身份。

痛点与弊端

传统的 Session 认证方式主要存在以下弊端：

  1. 开销大，服务端每进行一个用户的认证，就要为其保存一次 Session（通常保存在内存），随着用户数量越来越大，信息越来越复杂，无疑增加了服务端的开销压力。
  2. 难扩展，在前后端分离和分布式集群项目中，项目通过会部署在多台服务器上，如果要实现信息同步，通常还需要实现 Session 的数据共享。
  3. CSRF，因为是基于cookie来进行用户识别的, cookie如果被截获，用户就会很容易受到跨站请求伪造的攻击。

JWT 的 Token 认证

JWT 简介

JWT 全称为 JSON Web Token，是一种基于 JSON 的通信开放标准，是在通信双方之间使用基于 JSON 格式的 Token 进行信息传输的一种方式，JWT 还使用了数字签名，所以使得这些信息安全可信。

JWT 主要运用与以下场景：

  1. 授权：用户成功登陆后，由服务端生成一个可信任的 Token，这个 Token 返回给客户端，后续的每一次请求，客户端都会携带这个 Token，服务端验证 Token 正确后，允许客户端进行对资源的访问。
  2. 信息传输：由于 JWT 可以签名，所以可以使用它安全地在各方之间进行信息的传输与交换，如 A 系统发送一些重要地信息给 B 系统，B 系统收到信息后，可以对其进行校验，确定是否可信以及是否被篡改。

JWT 的优点：

  1. 简洁轻快：JWT 数据量小，可以通过 URL，POST 和请求的 Header 域来发送，占用内存少；
  2. 自包含：JWT 中可以存储一些用户信息，减少与数据库的交互次数；
  3. 安全可信：由于签名算法的存在，使得信息安全可信；
  4. 扩展性高：无需保存会话信息，易于扩展，且可以跨语言。

JWT 的结构

JWT 的外表其实是一串形如 XXXX.YYYY.ZZZZ 的字符串，如 eyJhbGciOiJIUI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTgRG9lIiwiYWRtaW4iOnRydWV9.TJVA9cBab30RMHrHDcFh7HgQ，字符串之间用 . 分开，其实表示了三个部分的信息：

  1. Header：头部信息
  2. Payload：负载信息
  3. Signature：签名信息

这三个部分通过一定的编码或加密算法后，形成了上述的字符串。

Header

一个头部信息主要由两个部分组成：

  1. Token 类型- typ。
  2. 签名算法- alg，如HMACSHA256或RSA。
