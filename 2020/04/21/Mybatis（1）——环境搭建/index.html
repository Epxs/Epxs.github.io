Mybatis(1)——环境搭建

发表于 2020-04-21 | 分类于 Mybatis

MyBatis 是一款优秀的持久层框架，它支持自定义 SQL、存储过程以及高级映射。MyBatis 免除了几乎所有的 JDBC 代码以及设置参数和获取结果集的工作。

JDBC 回顾

使用 JDBC 存在以下缺点：

  • 造成代码冗余，相同代码重复出现，降低了程序可读性。
  • 当数据库字段较多时，解析数据困难，且容易出错。
  • sql 代码和 java 代码存在耦合。
  • 频繁的开启和关闭资源，造成资源浪费。

Mybatis 简介

Mybatis 是一个 ORM 持久层框架，支持定制化 SQL、存储过程以及高级映射。简单来讲，可以将 Mybaits 看成是对 JDBC 的封装，可以帮助开发者更加便捷的完成数据库交互。

使用 Mybatis 有以下优点：

  • 简单易学：本身就很小且简单。易于学习，易于使用。
  • 操作灵活：mybatis 不会对应用程序或者数据库的现有设计强加任何影响。 sql 写在 xml 里，便于统一管理和优化。
  • 解除 sql 与程序代码的耦合：将业务逻辑和数据访问逻辑分离，使系统的设计更清晰，更易维护，更易单元测试。sql和代码的分离，提高了可维护性。
  • 提供xml标签，支持编写动态sql。

Mybatis 工作流程

Mybaits 有三个重要的类，分别是：

  • SqlSessionFactoryBuilder
  • SqlSessionFactory
  • SqlSession

详细工作流程如下：

  1. 加载 mybatis 全局配置文件（数据源、mapper 映射文件等），解析配置文件，MyBatis 基于 XML 配置文件生成 Configuration，和一个MappedStatement。
  2. SqlSessionFactoryBuilder 通过 Configuration 对象生成 SqlSessionFactory，用来开启 SqlSession。
  3. SqlSession 对象完成和数据库的交互：
    • 用户程序调用 mybatis 接口层 api（即 Mapper 接口中的方法）
    • SqlSession 通过调用 api 的 Statement ID 找到对应的 MappedStatement 对象
    • 通过 Executor（负责动态SQL的生成和查询缓存的维护）将 MappedStatement 对象进行解析，sql 参数转化、动态 sql 拼接，生成 jdbc Statement 对象
    • JDBC 执行 sql。
  4. 借助 MappedStatement 中的结果映射关系，将返回结果转化成 HashMap、JavaBean 等存储结构并返回。

搭建第一个 Mybatis 程序

1 创建 Maven 项目，在 pom.xml 中添加相关依赖。

2 配置 mybatis-config.xml 文件，XML 配置文件中包含了对 MyBatis 系统的核心设置，包括数据源以及事务等。

3 创建一张数据表，并插入几条数据。

4 创建实体类 Student，对应 student 表相对应。

5 创建 dao 层接口。

6 根据 StudentMapper 接口配置 Studentmapper.xml 文件，该文件用来配置对应的 sql 语句。

7 配置好 mapper 文件后，还需要将 mapper 配置到 mybatis-config.xml 文件中注册，将 mapper 标签写在 configuration 标签之内即可。

在上面的测试方法中，我们是通过 SqlSession 去执行已经配置好的 sql 语句的，此外 mybatis 还提供了另外一种方法——使用和指定语句的参数和返回值相匹配的接口（如 StudentMapper.getAllStudent() ）。实现方式是通过 SqlSession 获取一个 Mapper 对象。

两种方式执行结果一致，但在实际开发中，更倾向于使用第二种方式，这样可以不仅可以使代码更清晰，更加类型安全，还不用担心可能出错的字符串字面值以及强制类型转换，也更符合设计思路。

Mybatis 框架实现了 sql 语句和 java 代码的分离，和 JDBC 相比，减少了代码量（尤其是对数据量大，业务复杂的场景而言），让开发者无需纠结与数据的解析与封装，只需关注 sql 语句的编写和业务方法的执行，使得数据库交互更加高效，更加便捷。
