Mybatis(3)——CRUD

发表于 2020-04-23 | 分类于 Mybatis

Mybatis 将 sql 写在 xml 文件中，通过 xml 映射实现对数据库的访问。

配置文件

Mybatis 的 config 文件中包含了 mybaits 的核心配置，通过该表属性的值，可以深深影响 Mybaits 行为的设置的属性信息。配置文档的结构如下：

  • configuration（配置）
    • properties（属性）
    • settings（设置）
    • typeAliases（类型别名）
    • typeHandlers（类型处理器）
    • objectFactory（对象工厂）
    • plugins（插件）
    • environments（环境配置）
      • environment（环境变量）
      • transactionManager（事务管理器）
      • dataSource（数据源）
    • databaseIdProvider（数据库厂商标识）
    • mappers（映射器）

配置 xml 文档必须遵循顺序，否则将报错。

properties 属性

properties 可以用来配置一些属性，属性可以在外部进行配置，也可以在 properties 元素的子元素中设置。设置好的属性可以在整个配置文件中用来替换需要动态配置的属性值。

1 在 resource 包下创建一个 db.properties 文件。

2 在 mybatis-config.xml 文件中通过 properties 引入外部配置文件后，便可以对某些属性进行替换。

typeAliases 类型别名

类型别名可为 Java 类型设置一个缩写名字。它仅用于 XML 配置，可以降低冗余的全限定类名书写。

3 设定注解后，在扫描包时，会使用注解中的值。

environments 环境

MyBatis 可以配置成适应多种环境，这种机制有助于将 SQL 映射应用于多种数据库之中， 现实情况下有多种理由需要这么做。例如，开发、测试和生产环境需要有不同的配置；或者想在具有相同 Schema 的多个生产数据库中使用相同的 SQL 映射。还有许多类似的使用场景。

关键点:

  • 默认使用的环境 ID（比如：default=”development”）。
  • 每个 environment 元素定义的环境 ID（比如：id=”development”）。
  • 事务管理器的配置（比如：type=”JDBC”）。
  • 数据源的配置（比如：type=”POOLED”）。

环境可以随意命名，但务必保证默认的环境 ID 要匹配其中一个环境 ID。

映射器 mappers

Mapper 可以告诉 MyBatis 到哪里去找到 sql 语句。有多种引入资源的方式。

mapper 文件中会存放已经定义好的 sql 语句，而这些配置便会告诉 Mybatis 该去哪里找到这个配置文件。

mapper 映射文件和 dao 层接口里面的方法一一对应，在 mapper 文件中，我们只需专注于写 sql 和配置相应的结果集。实现了 sql 语句和 java 代码的解耦，也是进行数据库交互的核心。

实现 CRUD

数据库交互最基本的便是增、删、改、查四个操作了，有了 Mybatis 后，这些操作也变得简单起来，只需写好接口方法，并在 mapper 文件去实现便可以了。

增删改查操作在 mapper 文件也对应着四个标签：

  • insert：映射插入语句
  • update：映射更新语句
  • delete：映射删除语句
  • select：映射查询语句

它们有几个共同的属性：

  • id：一般指向接口的方法名。

  • parameterType：指定需要传入的参数。

  • select 标签还可以设定方法的返回值类型，resultType。

注意：增、删、改操作必须提交事物。
